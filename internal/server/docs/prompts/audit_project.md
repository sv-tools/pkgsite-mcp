Audit the Go project{{if .Path}} at {{.Path}}{{else}} in the current directory{{end}} and all of its dependencies — including test-only dependencies, which run on CI — for security and maintenance risk. The pkg.go.dev tools (get_vulnerabilities, get_module, list_module_versions) are the primary source; govulncheck, when available, is an additional reachability check, not the main scanner.
1. Enumerate the modules and their selected versions with "go list -m all" if a Go toolchain is available (it covers build and test dependencies), otherwise read go.mod and go.sum{{if .Path}} under {{.Path}}{{end}}. When the toolchain is available, also run "go list -deps -test ./..." to learn which modules are actually imported (by build or test code) versus only present in the graph.
2. Known vulnerabilities: for each module at its selected version, call get_vulnerabilities to list advisories from the Go vulnerability database. A "not found" (404) means that version could not be checked — common for pseudo-versions (untagged "v0.0.0-<commit>") — so treat it as unknown, not clean. For each advisory, find the fixed version; if get_vulnerabilities omits it, call list_module_versions and re-check newer versions for the first clean one. Additionally, if govulncheck is available, run "govulncheck -test ./..."{{if .Path}} in {{.Path}}{{end}} to see which advisories are reachable from your code or tests ("-test" covers test code and test-only deps; if it fails to load because the tests do not compile, retry without "-test" and say so). Use its reachability as extra signal on top of the pkg.go.dev results, not as a replacement.
3. Unmaintained modules: call get_module for each module to read its latest version and commit time. Flag modules whose latest release is old (roughly two years or more) or that are several releases behind the version you use. If the toolchain is available, "go list -m -u all" also lists available updates and any author-deprecated module — report deprecated modules prominently, as that is the clearest "no longer supported" signal.
4. Unused modules: prefer "go mod tidy -diff"{{if .Path}} in {{.Path}}{{end}} — the require entries it would remove are the genuinely unused ones, since it keeps modules still needed transitively. If that is unavailable (older Go without -diff, or no toolchain), approximate by flagging modules in the graph that no package imports — missing from "go list -deps -test ./...", or where "go mod why -m <module>" reports "main module does not need" — but note that this over-reports legitimate transitive dependencies. Recommend "go mod tidy" to drop the unused requirements.
Then report, in this order:
- A one-line summary, e.g. "Scanned 40 modules: 2 vulnerable, 3 unmaintained, 5 unused."
- Vulnerabilities: a table of only the affected modules (module, version, advisory ID, summary, fixed version, and — if govulncheck ran — whether your code or tests reach it). Skip if none.
- Unmaintained: a short table (module, version, latest, latest release date, reason: deprecated / stale / behind). Skip if none.
- Unused: a list of modules removable with "go mod tidy". Skip if none.
- The minimal upgrades and removals you recommend, prioritizing reachable vulnerabilities.
If govulncheck was not available, note that vulnerability results reflect advisories by module and version only, not reachability, and do not cover the Go standard library or toolchain.
